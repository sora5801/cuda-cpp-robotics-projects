# Demo — 01.15 Background subtraction for fixed-workspace cells

One command builds (if needed), runs on the committed sample, and verifies the output:

```powershell
.\run_demo.ps1          # Windows (the required path: VS 2026 + CUDA 13.3)
```

```bash
./run_demo.sh           # Linux/macOS best-effort bonus (CMake path)
```

## What the demo demonstrates

The demo loads the committed 160-frame, 128x96 synthetic work-cell sequence and runs **three**
background-subtraction models — frame differencing (the naive baseline), a running single Gaussian
per pixel, and MOG-lite (a 3-mode Gaussian mixture per pixel) — on **both** the GPU and an
independent CPU oracle, cleans every raw mask with a shared 3x3 morphological open, verifies
GPU==CPU, then runs **five independent gates** that grade each model's OWN output against ground
truth recomputed directly from the sequence's designed event schedule (never through the GPU/CPU
twin comparison itself — see `../src/main.cu`'s file header and `THEORY.md` "How we verify
correctness" for why that independence matters).

What a learner should notice:

- **`GATE illumination_drift`**: frame-diff's false-positive rate explodes late in the sequence
  (after the +15% illumination ramp has accumulated) while both adaptive models stay low — the
  headline lesson of comparing a static reference against an adaptive one.
- **`GATE bimodal_lesson`**: at the blinking status-lamp panel, the single-Gaussian model's
  false-positive rate is *always* high (it cannot represent two legitimate brightness states at one
  pixel) while MOG-lite's stays low (it learns BOTH states as separate modes) — the reason this
  project builds a 3-mode mixture at all, not just a fancier single Gaussian.
- **`GATE absorption`**: the CSV artifact plots the single-Gaussian model's own foreground-fraction
  over the newly-placed box, alongside the CLOSED-FORM prediction of how many frames it should take
  to be absorbed (derived in `THEORY.md` "The math") — measured and predicted land within a 2x
  factor of each other on the reference machine.

Artifacts land in `demo/out/` (git-ignored — regenerated by every run):

| File | What it shows |
|------|----------------|
| `mask_strip_frame035.ppm`, `_frame065.ppm`, `_frame140.ppm` | Five panels side by side — original frame, ground-truth mask, frame-diff mask, single-Gaussian mask, MOG-lite mask — for three key moments (mid-E1 intrusion, just after the box is placed, mid-E5 intrusion after drift). |
| `absorption_curve.csv` | Per-frame foreground fraction over the box region, single-Gaussian and MOG-lite, plus the predicted absorption frame count in its header comment. |
| `fp_rate_timeline.csv` | Per-frame false-positive rate (over the "clean background" pixel set — README/THEORY.md define it) for all three models across the whole sequence. |
| `gates_metrics.csv` | One row per gate metric: value, threshold, verdict — the machine-readable twin of the `GATE ...:` lines below. |

Open any `.ppm` with an image viewer that supports the format (IrfanView, GIMP, or convert with
ImageMagick/Pillow to PNG); the CSVs open in Excel/any spreadsheet or plotting tool.

## How to read the output

| Line prefix | Meaning | Checked against `expected_output.txt`? |
|-------------|---------|----------------------------------------|
| `[demo]`    | Which project/demo this is. | Yes — stable. |
| `[info]`    | GPU name, and every MEASURED number (verify diffs, gate metrics) — varies by machine/GPU. | No. |
| `PROBLEM:`  | The exact problem instance (sequence length, frame size). | Yes — stable (demo runs with no args). |
| `DATA:`     | Confirms the 160-frame sequence loaded. | Yes — stable. |
| `VERIFY(<model>):` | `PASS`/`FAIL` — does the GPU's raw mask + final model state agree with the independent CPU oracle within documented tolerance, for that model? | Yes — stable. |
| `GATE <name>:` | `PASS`/`FAIL` — does that gate's independent, ground-truth-based check hold? See `../src/main.cu`'s gate functions and `THEORY.md` for each derivation. | Yes — stable. |
| `ARTIFACT:` | Confirms a `demo/out/` file was written (relative path only — the RESOLVED absolute path is never in a stable line, since it is machine-specific). | Yes — stable. |
| `[time]`    | CPU reference ms, GPU ms (325 kernel launches total), and a speed-up figure — a **teaching artifact, never a benchmark claim** (single-shot; first launches pay one-time init costs). | No. |
| `RESULT:`   | Overall `PASS`/`FAIL` — every `VERIFY` and every `GATE` must pass. The program exits nonzero on `FAIL`. | Yes — stable. |

The runner scripts do a **subset diff**: every non-comment line of
[`expected_output.txt`](expected_output.txt) must appear verbatim in the output; extra lines
(timings, device info) are allowed. `#`-prefixed lines in that file are comments.

## If it fails

- **Build fails:** see [`../../../docs/BUILD_GUIDE.md`](../../../docs/BUILD_GUIDE.md) (toolchain
  install, CUDA/VS integration, GPU architecture list).
- **`RESULT: FAIL`:** the GPU result disagreed with the CPU oracle — a real bug. Start in
  `../src/kernels.cu` and compare against `../src/reference_cpu.cpp`.
- **Expected-line mismatch only:** the program passed its own check but printed different stable
  lines — someone changed the output without updating `expected_output.txt` (or vice versa). The two
  are a contract; fix them together.
