# Data — 01.20 Time-of-flight raw processing: phase unwrapping, flying-pixel removal

Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh` — idempotent, with source URL, expected size, and checksum documented below. **Respect every
  license**; registration-gated or no-redistribution datasets are pointed at, never mirrored.
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

### Why no public dataset

Public iToF datasets do exist (e.g. academic depth-completion or SLAM corpora built on Kinect-v2 /
Azure Kinect captures), but every one of them ships already-DECODED depth/IR images from the vendor's
own closed on-chip pipeline — not the raw 4-tap correlation frames this project's decode math actually
operates on, and never with an independent, per-pixel "was this a genuine phasor mixture" ground-truth
label (the exact thing the `flying_pixel` gate needs to score precision/recall honestly). Loading one
here would mean grading against depth the SENSOR VENDOR already unwrapped and filtered — exactly the
computation this project teaches, done invisibly, upstream, by someone else's silicon. A rendered
synthetic scene, by contrast, gives EXACT ground truth (metric depth, surface identity, and true
sub-pixel flying-pixel membership — no photograph or vendor SDK could) for every one of this project's
nine gates. `../scripts/download_data.ps1`/`.sh` are honest no-ops recording this decision.

## Fields / format

### Tap stack (8 files, `data/sample/tof_f{1,2}_tap{0..3}.pgm`)

Every file is a binary PGM (P5), `160 x 120` pixels (== `kCamW x kCamH` in `../src/kernels.cuh`),
8-bit grayscale (`maxval 255`), row-major. These are the RAW CORRELATION TAP intensities (ambient +
signal + sensor noise already baked in — see "How the sample was tuned" below) — exactly what a real
iToF sensor's 4 charge-storage wells would hand a decode pipeline, one PGM per (frequency, tap) pair.

| Files | Count | Meaning |
|-------|-------|---------|
| `tof_f1_tap{0..3}.pgm` | 4 | Frequency 1 (60 MHz, the FINE channel), taps 0/90/180/270 degrees. |
| `tof_f2_tap{0..3}.pgm` | 4 | Frequency 2 (20 MHz, the COARSE channel), taps 0/90/180/270 degrees. |

### Ground truth (3 files, `data/sample/truth_*.bin`)

Raw little-endian binaries, row-major, one value per camera pixel (`19,200 = 160 x 120` values), **no
header** — the format is fixed by `kCamW`/`kCamH` in `kernels.cuh`, documented once here.

| File | Type | Units / frame | Meaning |
|------|------|------|---------|
| `truth_depth.bin` | `float32[19200]` | meters, camera frame | The true depth `Z` of the surface point the pixel CENTER ray sees (a single, non-supersampled ray — every camera ray hits SOME analytic surface in this scene, so there is no sentinel). |
| `truth_surface.bin` | `uint8[19200]` | enum, see `kernels.cuh` | Which analytic surface produced this pixel: `1` = background wall, `2` = sphere, `3` = box top. Used ONLY to grade the reconstruction gates in `main.cu` — never consumed by the decode/unwrap pipeline itself. |
| `truth_flying.bin` | `uint8[19200]` | boolean (0/1) | The INDEPENDENT flying-pixel ground truth: `1` iff `make_synthetic.py`'s SS x SS sub-ray supersampling found >= 2 distinct surfaces at this pixel with the second-largest surface's area weight >= 12.5% (2/16 sub-rays). Computed from generator-internal sub-ray membership the C++ detector never sees — see the script's file header for why this independence is load-bearing for the `flying_pixel` gate. |

### `params.csv`

Plain `key,value` rows (comment lines start with `#`). Two families of keys:

1. **Sensor/scene constants** that MUST equal the compiled `../src/kernels.cuh` values (`cam_w_px`,
   `cam_fx_px`, `freq1_hz`, `freq2_hz`, `num_taps`, `max_scene_depth_m`, …) — `main.cu`'s
   `assert_params_match()` checks every one of these on startup and fails loudly on any drift between
   this Python-generated file and the compiled C++ contract.
2. **Scene truth** used by the reconstruction gates: `sphere_radius_m_truth`, `step_height_m_truth`,
   the sphere/box/background geometry, the low-reflectivity dark-cohort pixel footprint
   (`dark_patch_row_min/max`, `dark_patch_col_min/max`), and per-surface pixel counts.

## How the sample was tuned

Every noise/albedo/threshold number in `../scripts/make_synthetic.py` and `kDefaultAmplitudeFloor` /
`kFlyingDepthJumpM` / `kFlyingAmplitudeRatio` in `kernels.cuh` was chosen by MEASURING the rendered
sample, not guessed — the repo's "measured-then-margined gate" convention (see e.g. project 01.19):

- **Modulation amplitude by surface** (intensity counts, `B` from the phase decode, freq1): background
  wall mean ~58 (min ~0.5 at silhouette-mixed pixels, max ~72), sphere mean ~53 (min ~17), box mean
  ~45 (min ~38), vs. the low-reflectivity dark cohort mean ~4.3 (max ~10.0). The gap between "normal
  surface" (>=17 even at its worst clean pixel) and "dark cohort" (<=10 everywhere) is what makes a
  single amplitude floor viable at all.
- **Amplitude floor = 18 counts** sits inside that gap: on the committed sample it rejects **100%** of
  the 1,200-pixel dark cohort (zero survivors, so the dark_cohort gate's "zero hallucinated survivors"
  condition holds trivially) while every legitimate surface's minimum clean-pixel amplitude (~17 for
  a few sphere-silhouette pixels, ~38 for the box) stays close to or above the floor — the residual
  handful of near-floor sphere pixels are themselves near the sphere's own silhouette (partial-coverage
  phasor mixing, the SAME mechanism the flying-pixel detector targets) and their exclusion is honest,
  not a defect.
- **Flying-pixel thresholds** (`kFlyingDepthJumpM = 0.30` m, `kFlyingAmplitudeRatio = 0.60`) were swept
  against the 108 true-flying pixels the generator's supersampling identifies (0.56% of the image):
  because every genuine edge in this scene (box/sphere against the far wall) has neighbor depth jumps
  of 1.5-4 m — far beyond any plausible `kFlyingDepthJumpM` choice in the 0.15-0.30 m range — the
  detector's PRECISION/RECALL trade-off is set entirely by the amplitude-ratio threshold. Sweeping it
  from 0.55 to 0.85 traced a clean precision/recall frontier (0.55 -> 100%/54%, 0.60 -> 100%/62%,
  0.65 -> 75%/71%, 0.85 -> 30%/98%); `0.60` was picked as the last point with ZERO false positives
  (no clean interior pixel misflagged) while still catching a majority of true flying pixels — a
  deliberately conservative, high-precision operating point (README "Limitations" states the
  recall/precision trade-off honestly; a learner sweeping the same threshold reproduces this table).
- **Per-tap noise sigma = 3.0 counts** (photon-shot + read noise, combined into one Gaussian per the
  file header's radiometric model) was chosen so the dual-frequency unwrap's wrap-count decision is
  correct on the large majority (~98%) of confident pixels but NOT 100% — a few genuine wrap failures
  (~2%) survive on the committed sample, each contributing a gross (~1 ambiguity range) depth outlier.
  This is deliberate and honest: THEORY.md derives the wrap-decision failure probability from first
  principles, and a noise level that made failures vanish entirely would teach nothing about why
  dual-frequency unwrapping is a NOISE-SENSITIVE decision, not a free lunch.
- All numbers above are reproduced live by the demo itself on every run (`[info]` lines) — this table
  records the TUNING rationale, not a frozen snapshot; regenerating the sample with a different seed
  will shift the exact figures while the gates (set with margin below/above what was measured) should
  still pass.
