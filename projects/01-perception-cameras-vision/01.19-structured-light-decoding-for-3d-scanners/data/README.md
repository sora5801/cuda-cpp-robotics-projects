# Data — 01.19 Structured-light decoding (Gray code, phase shift) for 3D scanners

Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh` — idempotent, with source URL, expected size, and checksum documented below. **Respect every
  license**; registration-gated or no-redistribution datasets (KITTI, nuScenes) are pointed at, never
  mirrored.
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

### Why no public dataset

Real structured-light capture sets exist (e.g. university-hosted Kinect-v1-era corpora) but every
one of them is calibrated for **its own specific camera-projector rig** — different intrinsics,
different baseline, different code parameters. Loading one here would either (a) require re-deriving
this project's `kernels.cuh` scanner-geometry constants from an unfamiliar calibration file (real
research, out of scope for a decoding-focused project), or (b) silently mismatch the compiled
contract, producing wrong triangulation with no obvious cause. A ray-cast synthetic scene, by
contrast, gives **exact** ground truth (continuous sub-pixel projector column, depth, and surface
identity — no photograph could) for every one of the eight verification gates in `../src/main.cu` —
the pedagogically correct choice for a project about the DECODE MATH, not about denoising real
sensor data (that is 01.11's job). `../scripts/download_data.ps1`/`.sh` are honest no-ops recording
this decision.

## Fields / format

### Pattern stack (20 files, `data/sample/*.pgm`)

Every file is a binary PGM (P5), `200 x 150` pixels (== `kCamW x kCamH` in `../src/kernels.cuh`),
8-bit grayscale (`maxval 255`), row-major. These are the RAW CAPTURED intensities (ambient + noise +
mild blur already baked in — see "How the sample was tuned" below) — exactly what a real camera
frame grabber would hand a scanner's decode pipeline.

| Files | Count | Meaning |
|-------|-------|---------|
| `gray_b{0..6}_direct.pgm` | 7 | Gray-code bit plane `b` (0 = MSB … 6 = LSB), projector fully following the bit pattern. |
| `gray_b{0..6}_inverse.pgm` | 7 | The SAME bit plane, photometrically inverted (the "captured + inverse" robustness trick — README "The algorithm in brief"). |
| `phase_s{0..3}.pgm` | 4 | 4-step phase-shift pattern, step `k` offset by `k*pi/2` (`kPhaseSteps` in kernels.cuh). |
| `ref_white.pgm` | 1 | Projector fully ON — a real-world calibration reference (`main.cu` uses it + `ref_black.pgm` only for a diagnostic dynamic-range `[info]` line, not gated). |
| `ref_black.pgm` | 1 | Projector fully OFF (ambient light only). |

### Ground truth (3 files, `data/sample/truth_*.bin`)

Raw little-endian binaries, row-major, one value per camera pixel (`30,000 = 200 x 150` values), **no
header** — the format is fixed by `kCamW`/`kCamH` in `kernels.cuh`, documented once here.

| File | Type | Units / frame | Meaning |
|------|------|------|---------|
| `truth_column.bin` | `float32[30000]` | projector columns, `[0, 128)` | The CONTINUOUS (sub-pixel) true projector column for this camera pixel's surface point. Sentinel `-1.0` marks pixels outside the projector's illuminated column range (a real, honest failure mode — see README "Limitations"). |
| `truth_depth.bin` | `float32[30000]` | meters, camera frame | The true depth `Z` of the surface point this camera pixel sees (always valid — every camera ray hits SOME analytic surface in this scene). |
| `truth_surface.bin` | `uint8[30000]` | enum, see `kernels.cuh` | Which analytic surface produced this pixel: `1` = background plane, `2` = sphere, `3` = box top. Used ONLY to grade the reconstruction gates in `main.cu` — never consumed by the decode/triangulation pipeline itself. |

### `params.csv`

Plain `key,value` rows (comment lines start with `#`). Two families of keys:

1. **Scanner/code constants** that MUST equal the compiled `../src/kernels.cuh` values
   (`cam_w_px`, `cam_fx_px`, `proj_cols`, `gray_bits`, `phase_period_cols`, `baseline_m`, …) —
   `main.cu`'s `assert_params_match()` checks every one of these on startup and fails loudly on any
   drift between this Python-generated file and the compiled C++ contract.
2. **Scene truth** used by the reconstruction gates: `sphere_radius_m_truth`, `step_height_m_truth`,
   the sphere/box/background geometry, the dark-albedo stripe's pixel footprint
   (`dark_stripe_row_min/max`, `dark_stripe_col_min/max`), and per-surface pixel counts.

## How the sample was tuned

Every noise/albedo/confidence-floor number in `../scripts/make_synthetic.py` and
`kDefaultConfidenceFloor` in `kernels.cuh` was chosen by MEASURING the rendered sample, not
guessed — the repo's "measured-then-margined gate" convention (see e.g. project 01.17):

- **Modulation amplitude by surface** (intensity counts, `B` from the phase decode): normal
  background ~48-70, sphere ~37-62, box ~33-57 (5th-percentile-to-max), vs. the dark-albedo stripe
  ~0-33 (median ~5) and pure sensor noise (no illumination at all) ~0-27 (median ~3). The gap between
  "normal surface" and "dark stripe / noise floor" is what makes a single threshold viable at all.
- **Confidence floor = 25 counts** was swept from 15 to 35: at 25, the dark-stripe cohort's
  rejection rate reaches 99.1% with **zero** surviving pixels reconstructing more than 60 mm from
  their true depth, while every legitimate surface's false-reject rate stays under ~2.5%. At the
  originally-tried floor of 15, rejection was still 90.6% but a handful (21 of 2,431 cohort pixels)
  of low-but-not-lowest-modulation survivors carried a WRONG Gray-decoded period (a bit-threshold
  error under low SNR — a mechanism the phase-modulation confidence signal does not directly guard
  against) and reconstructed 200+ mm off. This is a genuine, narrow blind spot of confidence-by-
  modulation-alone, not swept under the rug — see `THEORY.md` "Numerical considerations" and
  README "Limitations".
- **Boundary-stress noise sigma = 0.15** (in the stress test's own `[0,1]`-normalized units) was
  chosen so plain binary coding's boundary error rate (measured ~7.9%) is clearly, robustly larger
  than Gray code's (measured ~0.26%, a ~31x gap) without either rate collapsing to exactly 0% or
  saturating near 100% — the sweet spot where the SINGLE-BIT-ADJACENCY property both codes are being
  tested for is visibly, not just theoretically, decisive.
- All numbers above are reproduced live by the demo itself on every run (`[info]` lines in
  `demo/expected_output.txt`'s companion output) — this table records the TUNING rationale, not a
  frozen snapshot; regenerating the sample with a different seed will shift the exact figures while
  the gates (set with margin below/above what was measured) should still pass.
