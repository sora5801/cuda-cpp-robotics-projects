# Data — 02.03 Ground segmentation: RANSAC plane fit; Patchwork++-style GPU port

Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh` — idempotent, with source URL, expected size, and checksum documented below. **Respect every
  license**; registration-gated or no-redistribution datasets (KITTI, nuScenes) are pointed at, never
  mirrored.
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

## This project's data

This project's teaching goal — proving that a single RANSAC plane cannot represent sloped/multi-level
ground while a patch-local model can — requires a scene whose ground truth is *exact* and whose
geometry is *designed*: a flat segment, an 8° ramp, a raised plateau, standing obstacles, and a floating
canopy overhang, each analytically specified and ray-cast (no public dataset offers this on demand, and
none is needed — see `../scripts/download_data.ps1` for why this project has no download path).

## What the scene is

A 16-beam spinning-LiDAR scan (elevation −15°…+15° in 2° steps, cited verbatim from 01.18 via 02.01's
precedent; 2,560 azimuth steps/revolution, 12 accumulated revolutions — see the generator's module
docstring for why more revolutions are needed here than in 02.01) cast against an analytic scene with
**three ground levels**:

- a **FLAT** segment at `z = -1.5 m` (the sensor's mount height) covering most of the scene;
- an **8° RAMP** inside a narrow forward corridor (`4 m ≤ x < 8 m`, `|y| ≤ 3.5 m`), rising linearly;
- a **PLATEAU**: flat ground again, but raised by the ramp's rise (`≈ 0.562 m`), filling the corridor
  out to `x = 12 m` (a documented "ledge" — nothing beyond it).

Deliberately **no room-bounding walls** (a departure from 02.01's enclosed room — the module docstring's
"Why no bounding walls" explains the measured problem this fixes: with ~half this beam table's
elevations pointing upward, room walls dominated the point count and could out-vote the true ground
plane in RANSAC's own hypothesis pool). Instead: **six standing obstacles** (two boxes + one pole on the
flat segment, one box + one pole on the plateau, and one thin wall-*segment* — the catalog's "...a
wall", modeled as an obstacle, not a room boundary) and one **canopy overhang** (a disc of 2,500 points
floating 1.8–3.1 m above flat ground, deliberately positioned to overlap a real ground-return ring — see
the module docstring's discussion of this scene's discrete-range beam-ring structure).

Measured composition of the committed file (an actual `make_synthetic.py` run, not estimated):
491,520 candidate rays cast (16 beams × 2,560 azimuth steps × 12 revolutions) → **159,336 beam-scan
returns** (32.4% hit rate; the rest are honest dropouts — most upward-pointing beams and any beam
whose ground/obstacle intersection lies beyond `MAX_RANGE_M`), of which 135,972 are true ground
(105,828 flat + 27,540 ramp + 2,604 plateau) and 23,364 are box/pole/wall-segment obstacle returns,
plus 2,500 appended canopy points = **161,836 points total**.

### Fields / format (`ground_scan.bin`)

A small hand-rolled binary format (no library — CLAUDE.md §5's "no black boxes" stance), read by
`../src/main.cu`'s `load_ground_scan()` with **explicit fixed-width primitive reads** (never a raw
struct `fread`, so the layout never depends on a compiler's struct-padding rules):

| Bytes | Type | Field | Meaning |
|-------|------|-------|---------|
| 0–7 | `char[8]` | magic | literal `GNDSEG01`, no null terminator |
| 8–11 | `int32` | `n_total` | total point count |
| 12–15 | `int32` | `n_flat` | true-ground flat-segment points |
| 16–19 | `int32` | `n_ramp` | true-ground ramp points |
| 20–23 | `int32` | `n_plateau` | true-ground plateau points |
| 24–27 | `int32` | `n_nonground_beam` | box/pole/wall-segment (non-ground) returns in the beam scan |
| 28–31 | `int32` | reserved | always 0 |
| 32–35 | `int32` | `n_canopy` | appended canopy/overhang points (all non-ground) |
| 36–39 | `float32` | `sensor_height_m` | LiDAR mount height (1.5 m) — `main.cu` asserts this matches `kernels.cuh`'s `kSceneSensorHeightM` |
| 40–43 | `float32` | `ramp_slope_deg` | ramp grade (8.0°) — `main.cu` asserts this matches `kernels.cuh`'s `kSceneRampSlopeDeg` |
| 44–47, 48–51 | `int32` x2 | reserved | always 0 |
| 52 … | per point, 20 bytes each | `x,y,z,ground_label,zone_id` | see below, in the FIXED order [beam-scan points, cast order] then [canopy points] |

Per point (20 bytes): `float32 x, y, z` (meters, LiDAR **sensor frame** — origin at the sensor, +x
forward, +z up — CLAUDE.md §12; the same `PointCloud` convention 02.01/02.06 use, so this file's output
slots directly into those projects' inputs, and this project's own `point_ground` output is exactly
the mask 02.04/14.02 expect as an upstream input); `int32 ground_label` (1 = true ground, 0 = not);
`int32 zone_id` (0 = flat, 1 = ramp, 2 = plateau, −1 = not ground — obstacle or canopy).

> **Synthetic, labeled everywhere.** Every point in this file is analytically generated (ray-cast
> against a hand-specified ground/ramp/plateau/obstacle scene, or placed directly for the canopy
> overhang) — nothing here is a recording of any real sensor or real space. `main.cu`'s `DATA:` line
> and this file say so explicitly. Per-point ground truth (`ground_label`, `zone_id`) is *exact by
> construction* (the generator knows which analytic surface produced each return), not estimated.
