# Data — 02.15 Point cloud compression (octree/entropy) for fleet uplink

Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh` — idempotent, with source URL, expected size, and checksum documented below. **Respect every
  license**; registration-gated or no-redistribution datasets (KITTI, nuScenes) are pointed at, never
  mirrored.
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

## This project's data

Two committed clouds, both **synthetic**, both exactly 200,000 points — a deliberately fair,
apples-to-apples pair so the only variable between them is the *geometry*, which is exactly the
variable this project's compressibility story is about.

|----------|-------|
| Kind | Synthetic (repo default). Two point clouds: a structured warehouse-room map tile (surfaces) and a pathological uniform-random cube (no surfaces). |

| License | Synthetic — repo MIT license applies (no external data, no redistribution concerns) |
| Size (committed) | `sample/structured_map.bin` — 2,400,012 bytes (~2.29 MiB); `sample/pathological_cube.bin` — 2,400,012 bytes (~2.29 MiB) |

| Regenerate with | `python ../scripts/make_synthetic.py` (writes both files; add `--seed N` to experiment, but the checked-in samples and `demo/expected_output.txt` assume the default seed 42) |

### Why two clouds, and why they are built the way they are

- **`structured_map.bin`** — a small warehouse-room map tile: one floor rectangle, four wall
  rectangles (interior faces only — a fleet map is an aggregated multi-view reconstruction, not a
  single occluded scan), and five box-shaped obstacles (furniture/pallets), each contributing up
  to 5 visible faces (top + 4 sides). Points are allocated **proportional to surface area** (a
  constant areal point density — the same physical assumption a real dense map scan makes) and
  each carries a small (±3 mm) jitter along its surface's normal direction, so the geometry is a
  thin *slab* rather than a mathematically zero-thickness plane. This is the "surfaces are 2-D
  manifolds in 3-D space" half of the contrast — see `../THEORY.md` "The problem".
- **`pathological_cube.bin`** — 200,000 points drawn **uniformly at random** inside a cube whose
  side length matches the structured tile's own bounding cube (so the two clouds occupy comparable
  physical volumes at comparable densities) — geometry with **no surface structure at all**, the
  designed worst case `../src/main.cu`'s `GATE entropy_payoff` measures against.

Both clouds are generated by `../scripts/make_synthetic.py`, reusing the xorshift32 deterministic
RNG this repo's other point-cloud projects (02.01, 02.02) use, cited there.

### Binary format (both files, identical layout)

All fields little-endian, written with explicit `struct.pack` formats (never a raw struct dump), so
the layout is compiler-independent. `../src/main.cu`'s `load_cloud()` reads the identical sequence.

| Offset | Bytes | Field | Meaning |
|--------|-------|-------|---------|
| 0 | 8 | magic | `b'PCFU0001'` (no null terminator) |
| 8 | 4 | `int32` N | point count (200,000 in both committed files) |
| 12 | `N*3*4` = 2,400,000 | `float32[]` | interleaved `(x,y,z)`, meters, right-handed **map frame** (+z up; these are static map tiles, not sensor scans, so no sensor-relative forward/left convention is claimed — see `../../../docs/SYSTEM_DESIGN.md` item 3) |

Total: `8 + 4 + 2,400,000 = 2,400,012` bytes — matches both committed file sizes exactly.

### Fields / format — semantics

- **Units & frame:** meters, right-handed map frame, +z up (SYSTEM_DESIGN.md §3.2's convention,
  applied to a static map tile rather than a live sensor scan).
- **No timestamps, no intensity, no ring/azimuth fields** — this project's codec compresses pure
  **geometry** (xyz only); a real fleet-uplink payload would also carry intensity/semantic labels,
  which `../THEORY.md` "Where this sits in the real world" and `../PRACTICE.md` §3 discuss as a
  scoping decision, not an oversight.
- **`structured_map.bin`'s room geometry** (meters, map frame): a 20 m × 20 m room (floor `z=0`,
  walls `z∈[0,3]`), five axis-aligned obstacle boxes — all dimensions are hardcoded constants in
  `../scripts/make_synthetic.py` (`ROOM_HALF_M`, `WALL_HEIGHT_M`, `BOXES`), illustrative, not
  derived from any real warehouse layout.
- **`pathological_cube.bin`'s cube half-extent** is *measured* from the structured tile's own
  bounding cube at generation time (not hardcoded), so the two clouds always occupy comparable
  volumes even if the room geometry constants above ever change.
