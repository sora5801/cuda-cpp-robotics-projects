Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh` — idempotent, with source URL, expected size, and checksum documented below. **Respect every
  license**; registration-gated or no-redistribution datasets (KITTI, nuScenes) are pointed at, never
  mirrored.
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

## What the scene is

The same **16-beam spinning-LiDAR room scan** project [`02.01`](../../02.01-voxel-grid-downsampling-with-gpu-spatial-hashing/data/README.md)
pioneered in this domain (cited, reimplemented here — not imported, per CLAUDE.md's self-containment
rule): a 16 m × 16 m room (walls at `x,y = ±8 m`, no ceiling), a floor at `z = -1.5 m`, three obstacle
boxes, ray-cast by the 16-beam elevation table project [`01.18`](../../../01-perception-cameras-vision/01.18-depth-completion/THEORY.md)
derives, over 6 accumulated revolutions (245,760 candidate rays → 195,384 valid returns, 79.5% hit
rate). On top of that shared machinery, **this project adds its own two adversarial regions**,
parameterized for the KNN-vs-fixed-radius contrast this project teaches (different numbers than
02.01's own regions, which were tuned for hash-table stress instead):

- a **DENSE cluster** — 4,000 points in a 1.2 m cube at `(-6.0, 6.0, -1.0)`. A radius search at
  `r = 0.5 m` (`kernels.cuh`'s `kRadiusM`) centered here returns **1,215 neighbors** (measured) — the
  case fixed-radius voxel hashing is fast and correct for.
- a **SPARSE region** — 20 points on a 3 m grid (6× the search radius), floating 1 m above the floor,
  far from every wall/box/other point. A radius search at the SAME `r = 0.5 m` centered between two
  sparse cells returns **0 neighbors** (measured) — the physically-motivated failure a fixed radius
  hits somewhere in every real scan (THEORY.md "The problem" derives why from the sensor's 1/r² density
  falloff). KNN (K=8) returns exactly 8 neighbors at **both** locations — just farther away at the
  sparse one — which is this project's whole engineering point, made measurable.

**Queries.** 2,000 total, in three fixed, documented blocks (see `scripts/make_synthetic.py`'s
`build_queries()` docstring for the exact construction): 2 **designed** queries (the dense-cluster
center and a point between sparse cells — indices 0 and 1, read from the file header as
`idx_dense_query` / `idx_sparse_query`); 998 **self-query** points (a deterministic stride sample of
the point cloud itself); 1,000 **grid** queries (a systematic x/y grid at two heights, for broad,
unbiased spatial coverage). The first 1,000 queries (2 designed + all 998 self-query) are the
documented `n_anchor` subset `GATE brute_force_anchor` checks against an O(N·Q) linear-scan oracle.

### Fields / format (`lbvh_scan.bin`)

A small hand-rolled binary format (no library — CLAUDE.md's no-black-boxes stance), read by
`../src/main.cu`'s `load_sample()` with **explicit fixed-width primitive reads** (never a raw struct
`fread`, so the layout never depends on a compiler's struct-padding rules):

| Bytes | Type | Field | Meaning |
|-------|------|-------|---------|
| 0–7 | `char[8]` | magic | literal `LBVHSCN1`, no null terminator |
| 8–11 | `int32` | `n_points` | total point count |
| 12–15 | `int32` | `n_beam` | beam-scan points, occupy `xyz[0 .. n_beam)` |
| 16–19 | `int32` | `n_dense` | adversarial dense-cluster points, `xyz[n_beam .. n_beam+n_dense)` |
| 20–23 | `int32` | `n_sparse` | adversarial sparse-region points, the remaining `xyz[.. n_points)` |
| 24–27 | `float32` | `radius_m` | the search radius the sample was designed against — **must equal** `kernels.cuh`'s `kRadiusM` (0.5 m); `main.cu` asserts this at load time |
| 28–31 | `int32` | `n_queries` | total query count (2,000) |
| 32–35 | `int32` | `idx_dense_query` | query index of the dense-cluster designed query (0) |
| 36–39 | `int32` | `idx_sparse_query` | query index of the sparse-region designed query (1) |
| 40–43 | `int32` | `n_anchor` | how many leading queries `GATE brute_force_anchor` checks (1,000) |
| 44 … | `float32[n_points*3]` | `xyz` | interleaved x,y,z, **meters**, LiDAR **sensor frame** (origin at the sensor, +x forward, +z up — CLAUDE.md §12; the same `PointCloud` convention 02.01/02.06 use) |
| … | `float32[n_queries*3]` | `queries` | interleaved x,y,z, meters, sensor frame, in the three fixed blocks above |

> **Synthetic, labeled everywhere.** Every point and query in this file is analytically generated
> (ray-cast against a hand-specified room + boxes, or placed directly for the adversarial regions and
> the query grid) — nothing here is a recording of any real sensor or real space. `main.cu`'s `DATA:`
> line and every artifact this project writes say so explicitly.
