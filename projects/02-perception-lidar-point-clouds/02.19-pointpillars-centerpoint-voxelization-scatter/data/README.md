# Data — 02.19 PointPillars/CenterPoint voxelization + scatter kernels feeding TensorRT

Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored — none exists here).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh`. This project deliberately does **not** use one — see `../scripts/download_data.ps1`'s file
  header for why a real scan cannot supply this project's two load-bearing requirements (exact
  ground-truth object centers, and a precisely-placed cap-overflow pillar).
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

## What the scene is

An 80 m × 80 m bird's-eye-view window (the same window `src/kernels.cuh`'s `kXMin/kYMin/kGridNX/kGridNY`
define — `data/scene_meta.csv` is a data/code consistency contract, not a re-derivation), built directly
in the ego/BEV frame (origin at the sensor, +x forward, +y left, +z up — CLAUDE.md §12). Points are
written in **one fixed order**, back to back — the point-stream layout `scene_meta.csv` records and
`../src/main.cu` reads verbatim:

1. **Ground** (6,000 points) — a sparse, low-noise plane at z≈0, excluded under every car's footprint
   (a real LiDAR cannot see through a car — the same "honest dropout, not arbitrary thinning" philosophy
   project [`02.01`](../../02.01-voxel-grid-downsampling-with-gpu-spatial-hashing/data/README.md)'s
   generator uses for its walls' missing ceiling).
2. **6 cars** (220 points each, 1,320 total) — axis-aligned hollow boxes (roof + 4 side walls, no solid
   interior, no bottom face), 4.2 m × 1.8 m × 1.5 m, hand-placed at least 24 m apart (`CAR_CENTERS` in
   the generator) so no two objects can ever be confused by any tolerance this project uses.
3. **Clutter** (150 points) — low (z ≤ 0.35 m), spatially isolated single returns, rejection-sampled away
   from every car and the cap-stress pillar.
4. **Cap-stress pillar** (60 points, **deliberately > the 32-point-per-pillar cap** `kMaxPointsPerPillar`
   in `kernels.cuh`) — jittered inside exactly one pillar cell at BEV (−30.2, −30.2 m), low and narrow in
   z (0.2–0.5 m, never "tall" like a car) and spatially isolated (empty neighboring pillars) so the head
   cannot mistake it for an object. This is the fixture the `cap_truncation` gate studies — see
   `kernels.cuh`'s file header for the full determinism story it exists to expose.

Ground truth: `object_truths.csv` lists each car's BEV center, footprint, and height — the
`detection_closure` gate's answer key. `scene_meta.csv` records the point-stream composition counts and
the cap-stress pillar's exact key/point-index range, so `main.cu` never has to re-derive them by guessing.
`pfn_lite_weights.csv` holds the PFN-lite's **fixed, not-trained** linear-layer weights (seed 42,
continued from the same RNG stream as the scene) — see `kernels.cuh`'s and `THEORY.md`'s discussion of
why they are fixed and what a real trained PFN would do instead.

### Fields / format

**`points.bin`** — the plain **KITTI/PointPillars "raw velodyne.bin" layout** (no header at all): `N`
records of 4 little-endian `float32` back to back, `N` recovered from the file size (`N = bytes / 16`).
Using the exact on-disk shape of the real tool this project teaches toward is a deliberate, free bit of
authenticity (`THEORY.md` "Where this sits in the real world" names the real reference implementations).

| Field | Type | Meaning |
|-------|------|---------|
| x | `float32` | meters, BEV/ego frame, +x forward |
| y | `float32` | meters, BEV/ego frame, +y left |
| z | `float32` | meters, +z up |
| intensity | `float32` | unitless reflectance, sensor-reported, synthetic range [0.02, 0.9] here |

**`object_truths.csv`** (`#`-comment header, then one row per car):
`id,cx_m,cy_m,length_m,width_m,height_m,category` — `cx_m,cy_m` is the ground-truth BEV center every
`detection_closure` peak is measured against.

**`scene_meta.csv`** (`#`-comment header, then `key,value` rows) — the point-stream layout contract:
`n_total`, `n_ground`, `n_cars_total`, `n_cars`, `points_per_car`, `n_clutter`, `n_capstress`,
`capstress_start_index` (the point stream's first cap-stress point index), `capstress_pillar_key` (the
dense pillar-grid cell id `iy*kGridNX+ix` every 60 cap-stress points share), and the cap-stress pillar's
BEV center.

**`pfn_lite_weights.csv`** (`#`-comment header, then `kPfnLinOut`=4 rows): each row is
`w0,w1,...,w8,bias` — the fixed linear layer `kernels.cuh`'s `pfn_lite_kernel`/`pfn_lite_cpu` apply to
each kept point's 9-D augmented feature vector before ReLU + max-pool. **Not trained** — see the
generator's `build_pfn_weights()` docstring for the per-dimension weight-range reasoning.

> **Synthetic, labeled everywhere.** Every point in these files is analytically generated (placed
> directly by `scripts/make_synthetic.py`, seed 42, xorshift32) — nothing here is a recording of any
> real sensor or real space. `main.cu`'s `DATA:` line and every artifact this project writes say so
> explicitly.
