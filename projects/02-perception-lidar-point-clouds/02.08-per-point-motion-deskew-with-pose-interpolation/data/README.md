# Data — 02.08 Per-point motion deskew with pose interpolation

Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh` — idempotent, with source URL, expected size, and checksum documented below. **Respect every
  license**; registration-gated or no-redistribution datasets (KITTI, nuScenes) are pointed at, never
  mirrored.
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

## What the scene and cohorts are

A simplified version of [`02.01`](../../02.01-voxel-grid-downsampling-with-gpu-spatial-hashing/data/README.md)'s
room (cited): floor at `z = -1.5 m` and four walls at `x,y = ±8 m` spanning `z ∈ [-1.5, 1.5] m` (no
boxes — this project's payload is the *trajectory*, not scene clutter). A synthetic 16-beam spinning
LiDAR (the same −15°…+15°-in-2° elevation table [`01.18`](../../../01-perception-cameras-vision/01.18-depth-completion/THEORY.md)
derives and [`02.01`](../../02.01-voxel-grid-downsampling-with-gpu-spatial-hashing/scripts/make_synthetic.py)
cites) fires 360 azimuth steps (1°/step) across a single 100 ms sweep — a realistic 10 Hz spin rate.

Four **motion cohorts** ray-cast the identical static scene while the platform follows a different
**ground-truth trajectory** during the sweep (`../scripts/make_synthetic.py`'s module docstring derives
each one; `THEORY.md` "The problem" explains the physics):

| # | Cohort | Ground-truth motion | What it teaches |
|---|--------|---------------------|------------------|
| 0 | `straight` | constant velocity 15 m/s, constant heading | position LERP is *exact* here — the consistency check between the dense and sparse regimes |
| 1 | `arc` | constant yaw rate 6 rad/s (~34° over the sweep), 15 m/s forward | orientation SLERP is exact for constant angular rate; 2-sample POSITION interpolation is a chord across the true arc — a measurable bias |
| 2 | `wiggle` | 10 m/s translation + a 25 Hz sinusoidal yaw oscillation (~2.5 cycles inside the sweep), ±20° amplitude | the designed cohort: 2-sample interpolation cannot see oscillations that happen entirely between its two samples |
| 3 | `stationary` | zero velocity, zero yaw rate | the identity control — a correct deskew must be a no-op |

For every candidate (azimuth step, beam) pair, the generator casts **one** ray from the platform's TRUE
pose at that ray's own firing time. The hit yields two records for the *same physical point*: the
**skewed** record (local coordinates at firing time — raw driver output) and the **truth** record (that
identical point, analytically re-expressed in the reference frame via the platform's TRUE, continuous
pose — see the generator's `build_cohort_points` docstring for why this, and *not* a second independent
ray-cast in the same local direction, is the correct oracle). Measured composition of the committed file
(an actual `make_synthetic.py` run, not estimated): 5,760 candidate rays per cohort (16 beams × 360
azimuth steps) → 4,605 / 4,635 / 4,606 / 4,580 valid paired points for straight/arc/wiggle/stationary
respectively (≈80% hit rate; the rest are honest dropouts — beams that clear the wall top before
reaching a wall, since the scene has no ceiling).

### Fields / format (`deskew_scan.bin`)

A small hand-rolled binary format (no library — CLAUDE.md §1's no-black-boxes stance), read by
`../src/main.cu`'s `load_sample()` with **explicit fixed-width primitive reads** (never a raw struct
`fread`, so the layout never depends on a compiler's padding/alignment rules):

**Header** (32 bytes):

| Bytes | Type | Field | Meaning |
|-------|------|-------|---------|
| 0–7 | `char[8]` | magic | literal `DESKEW01`, no null terminator |
| 8–11 | `int32` | `num_cohorts` | 4 |
| 12–15 | `int32` | `num_beams` | 16 — **must equal** `kernels.cuh`'s `kNumBeams`; `main.cu` asserts this at load time |
| 16–19 | `int32` | `azimuth_steps` | 360 — must equal `kernels.cuh`'s `kAzimuthSteps` |
| 20–23 | `int32` | `n_dense_samples` | 21 — must equal `kernels.cuh`'s `kDenseSamples` |
| 24–27 | `float32` | `sweep_duration_s` | 0.100 (100 ms) |
| 28–31 | `float32` | `room_half_m` | 8.0 (documentation only) |

**Then, for each of the 4 cohorts, IN FIXED ORDER** (straight=0, arc=1, wiggle=2, stationary=3 —
`kernels.cuh`'s `cohort_name()` documents this order):

| Field | Type | Meaning |
|-------|------|---------|
| `cohort_id` | `int32` | 0..3, redundant with position — `main.cu` checks it |
| `n_points` | `int32` | paired skewed/truth point count for this cohort |
| dense trajectory | `float32[n_dense_samples*8]` | `(t_s, px,py,pz, qw,qx,qy,qz)` per sample, WORLD frame, repo quaternion order (w,x,y,z) — `kernels.cuh`'s "TRAJECTORY LAYOUT" |
| per point × `n_points` | `float32 t_s`, `int32 beam_id`, `float32 x,y,z` | the **SKEWED** point: firing time (s), beam index (0..15, indexes `kBeamElevDeg`), local xyz (m) in the sensor's own frame AT `t_s` |
| per point × `n_points` | `float32 x,y,z` | the **TRUTH** point: the same physical point, meters, in the reference frame (see "What the scene and cohorts are" above) |

> **Synthetic, labeled everywhere.** Every point in this file is analytically generated (ray-cast
> against a hand-specified room, with the platform following a hand-specified trajectory) — nothing
> here is a recording of any real sensor, IMU, or real space. `main.cu`'s `DATA:` line and this file say
> so explicitly.
