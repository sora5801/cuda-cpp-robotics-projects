# Data — 02.01 Voxel-grid downsampling with GPU spatial hashing

Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh` — idempotent, with source URL, expected size, and checksum documented below. **Respect every
  license**; registration-gated or no-redistribution datasets (KITTI, nuScenes) are pointed at, never
  mirrored.
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

## What the scene is

A 16 m × 16 m room (walls at `x,y = ±8 m`, no ceiling), a floor at `z = -1.5 m` (sensor mounted
1.5 m up), and three obstacle boxes standing on the floor — all analytic geometry, ray-cast by a
synthetic 16-beam spinning LiDAR (elevation −15°…+15° in 2° steps, the beam model project
[`01.18`](../../../01-perception-cameras-vision/01.18-depth-completion/THEORY.md) derives and this
script cites verbatim). The scan accumulates **6 revolutions** of a realistic ~0.14°-azimuth-resolution
sweep (a documented scope choice — see the generator's module docstring — standing in for a short
integration dwell, exactly the scenario that motivates downsampling before a consumer like
[`02.06`](../../02.06-icp-point-to-point-point-to-plane-gicp/README.md) ICP ever sees the cloud).
Two **adversarial regions** are appended on top of the scan, deliberately, to stress the hash table in
opposite ways (see `THEORY.md` "The math" for the collision-probability arithmetic these exercise):

- a **DENSE cluster** — 3,000 points crammed into a 15 cm cube (smaller than one 20 cm voxel);
- a **SPARSE region** — 150 points on a 1 m grid (5× the voxel edge), each landing in its own voxel.

Measured composition of the committed file (an actual `make_synthetic.py` run, not estimated):
245,760 candidate rays cast (16 beams × 2,560 azimuth steps × 6 revolutions) → **195,384 valid
returns** (79.5% hit rate; the rest are honest dropouts — beams that clear the wall top before
reaching a wall, since the scene has no ceiling) + 3,000 dense-cluster points + 150 sparse-region
points = **198,534 points total**.

### Fields / format (`lidar_scan.bin`)

A small hand-rolled binary format (no library — see `THEORY.md` "The GPU mapping" for the repo's
general no-black-boxes stance), read by `../src/main.cu`'s `load_point_cloud()` with **explicit
fixed-width primitive reads** (never a raw struct `fread`, so the layout never depends on a compiler's
struct-padding rules):

| Bytes | Type | Field | Meaning |
|-------|------|-------|---------|
| 0–7 | `char[8]` | magic | literal `VXLSCAN1`, no null terminator |
| 8–11 | `int32` | `n_total` | total point count |
| 12–15 | `int32` | `n_normal` | beam-scan points, occupy `xyz[0 .. n_normal)` |
| 16–19 | `int32` | `n_dense` | adversarial dense-cluster points, `xyz[n_normal .. n_normal+n_dense)` |
| 20–23 | `int32` | `n_sparse` | adversarial sparse-region points, the remaining `xyz[.. n_total)` |
| 24–27 | `float32` | `leaf_m` | voxel edge length used to design the adversarial regions — **must equal** `kernels.cuh`'s `kVoxelLeafM` (0.20 m); `main.cu` asserts this at load time |
| 28–31 | `float32` | `sensor_height_m` | LiDAR mount height above the floor (1.5 m), documentation only |
| 32–35 | `int32` | `num_beams` | 16 (documentation only) |
| 36–39 | `int32` | reserved | always 0 |
| 40 … | `float32[n_total*3]` | `xyz` | interleaved x,y,z, **meters**, LiDAR **sensor frame** (origin at the sensor, +x forward, +z up — CLAUDE.md §12; the SAME `PointCloud` convention project [`02.06`](../../02.06-icp-point-to-point-point-to-plane-gicp/src/kernels.cuh) uses, so this file's output slots directly into that project's input) |

> **Synthetic, labeled everywhere.** Every point in this file is analytically generated (ray-cast
> against a hand-specified room + boxes, or placed directly for the two adversarial regions) — nothing
> here is a recording of any real sensor or real space. `main.cu`'s `DATA:` line and every artifact this
> project writes say so explicitly.
