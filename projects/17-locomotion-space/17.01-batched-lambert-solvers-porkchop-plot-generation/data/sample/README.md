# `data/sample/` — the tiny committed sample

This folder holds the **small, committed, offline-runnable** sample the demo consumes — the reason
anyone can clone this repo, build, and see the documented result with **zero downloads** (CLAUDE.md
§4, §8). Rules of the folder:

- **Tiny.** Kilobytes preferred, megabytes tolerated, ~50 MB is the hard ceiling (and means you
  should rethink). The demo must run fast and the repo must stay clonable.
- **Labeled.** Synthetic data says so; public data names its source and license — both in
  [`../README.md`](../README.md), which documents provenance, checksums, and every field's meaning,
  units, and frame.
- **Reproducible.** Everything here can be regenerated by `../../scripts/make_synthetic.py` (no RNG —
  a scenario is constants) or re-fetched by `../../scripts/download_data.ps1` — the committed bytes
  are a convenience, never the only copy of anything.
- **Self-contained.** If this project consumes data produced by a sensor-sim sibling project, the
  sample is **copied into** this folder — never referenced across project folders at build or run
  time (CLAUDE.md §4 self-containment rule).

**This project's sample:** `lambert_scenario.csv` (771 bytes, synthetic) — the task definition: two
circular heliocentric orbit radii (Earth-like at 1.000 AU, Mars-like at 1.524 AU), the shared
departure/arrival epoch window (28.0 TU), the accepted time-of-flight band (0.5–14.0 TU), and the
grid resolution (512×512). A Lambert solver's data *is* its scenario; there is no ephemeris to fetch
and no ground truth to store — correctness comes from the CPU rollout oracle (the §5 GPU-vs-CPU gate)
and the closed-form Hohmann check (verification against pure mathematics, `THEORY.md` §how-we-verify).
Format and checksum: [`../README.md`](../README.md); regenerate byte-identically with
`python ../../scripts/make_synthetic.py`.
