# Demo — 34.03 Ergodic control: spectral multiscale coverage (FFT-based — very GPU-friendly)

One command builds (if needed), runs on the committed sample, and verifies the output:

```powershell
.\run_demo.ps1          # Windows (the required path: VS 2026 + CUDA 13.3)
```

```bash
./run_demo.sh           # Linux/macOS best-effort bonus (CMake path)
```

## What the demo demonstrates

A single 2-D agent covers a synthetic two-hotspot information density over a 60 s / 6,000-step
closed loop, steered by Spectral Multiscale Coverage (SMC): the target's 1,024 Fourier coefficients
are computed once via a GPU DCT-via-cuFFT pipeline, then every control step updates all 1,024
modes in parallel (one GPU thread per mode) and steers by the resulting descent direction. Five
gates are checked (transform correctness, GPU-vs-CPU numerical agreement, ergodic-metric decrease,
basin-mass coverage, and a lawnmower negative control) — see `../README.md` §Expected output for
the actual measured numbers. Four artifacts are written to `out/` (git-ignored; regenerated by
every run):

- **`trajectory.csv`** — `t_s,x1,x2,u1,u2`, one row per control step. Plot `x1` vs. `x2`: the path
  should visibly *thicken* over the two hotspots relative to the empty background.
- **`ergodic_metric.csv`** — `t_s,epsilon`, one row per step. Plot on a log-y axis: a clear downward
  trend with the occasional small transient uptick (THEORY.md §numerics explains why that is
  expected, not a bug).
- **`target_phi.pgm` + `empirical_coverage.pgm`** — ASCII (P2) grayscale images, `24x24` pixels, the
  side-by-side comparison payload: the true target density vs. a coarse histogram of where the
  agent actually spent its time. **Each image is scaled independently to its own maximum** (not a
  shared absolute scale) — the smooth continuous target and the sparse ~10-samples/cell empirical
  histogram have very different peak-to-mean ratios, so a shared scale would either wash out the
  target or saturate the histogram to a wall of white. Look for **shape** agreement (two bright
  blobs in roughly the right places and relative sizes), not brightness-for-brightness agreement —
  the precise, absolute-mass comparison is the numerical COVERAGE gate, not this picture. View
  either file in any text editor (it is plain ASCII) or convert with any PGM-aware image tool.

## How to read the output

| Line prefix | Meaning | Checked against `expected_output.txt`? |
|-------------|---------|----------------------------------------|
| `[demo]`    | Which project/demo this is. | Yes — stable. |
| `[info]`    | GPU name, scenario path, and measured numbers (basin masses, deviations, epsilon, coverage fractions) — legitimately variable across GPU architectures even though this project has no RNG (see `THEORY.md` §Numerical considerations "Determinism"). | No. |
| `PROBLEM:` / `SCENARIO:` | The exact problem instance (mode count, horizon, dt, speed budget, start position). | Yes — stable (demo runs with no args). |
| `[time]`    | Setup/kernel timings — a **teaching artifact, never a benchmark claim** (single-shot; the first cuFFT plan call pays a one-time module-load cost). | No. |
| `TRANSFORM:` / `VERIFY:` / `ERGODICITY:` / `COVERAGE:` / `NEGATIVE-CONTROL:` | The five gate verdicts (tolerances documented in `../src/main.cu` and `THEORY.md`). | Yes — stable. |
| `ARTIFACT:` | Confirms all four output files were written. | Yes — stable. |
| `RESULT:`   | Overall `PASS`/`FAIL` — requires all five gates plus successful artifact writes. The program exits nonzero on `FAIL`. | Yes — stable. |

The runner scripts do a **subset diff**: every non-comment line of
[`expected_output.txt`](expected_output.txt) must appear verbatim in the output; extra lines
(timings, device info) are allowed. `#`-prefixed lines in that file are comments.

## If it fails

- **Build fails:** see [`../../../docs/BUILD_GUIDE.md`](../../../docs/BUILD_GUIDE.md) (toolchain
  install, CUDA/VS integration, GPU architecture list).
- **`RESULT: FAIL`:** the GPU result disagreed with the CPU oracle — a real bug. Start in
  `../src/kernels.cu` and compare against `../src/reference_cpu.cpp`.
- **Expected-line mismatch only:** the program passed its own check but printed different stable
  lines — someone changed the output without updating `expected_output.txt` (or vice versa). The two
  are a contract; fix them together.
