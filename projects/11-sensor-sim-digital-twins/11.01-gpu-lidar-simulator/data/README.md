# Data — 11.01 GPU LiDAR simulator: BVH raycasting + beam divergence, intensity, dropout noise

Provenance, licensing, and field documentation for everything under `data/` (CLAUDE.md §8).

## The rules (repo-wide, CLAUDE.md §8)

- **Synthetic-first.** Robotics data can almost always be synthesized with full ground truth;
  `../scripts/make_synthetic.py` is the default source, and synthetic data is **labeled synthetic
  everywhere it appears**.
- **Tiny committed sample.** `sample/` holds just enough committed data that the demo runs offline
  with zero downloads. Large/raw/downloaded data lives in `downloaded/` (git-ignored).
- **Public datasets** (where one genuinely teaches more) are fetched by `../scripts/download_data.ps1`
  / `.sh` — idempotent, with source URL, expected size, and checksum documented below. **Respect every
  license**; registration-gated or no-redistribution datasets (KITTI, nuScenes) are pointed at, never
  mirrored.
- **Never fabricate.** No made-up measurements, no invented ground truth passed off as real.

## This project's data

A LiDAR simulator's "dataset" is a **scene** (a triangle mesh) plus a **sensor spec** (scan pattern +
noise-model constants) — not recordings. Both are pure, RNG-free geometry/constants generated by
`../scripts/make_synthetic.py`; the beam-level RANDOMNESS (dropout draws, range jitter) is generated
*inside the demo itself* from the fixed seed in `sensor_config.csv`, exactly like 08.01's exploration
noise — so nothing stochastic needs to be committed at all.

| Property | Value |
|----------|-------|
| Kind | **Synthetic** — a hand-authored warehouse scene + sensor spec (no RNG anywhere in the generator; every byte is deterministic geometry/constants) |
| Generator | `python ../scripts/make_synthetic.py` (writes all four files below) |
| License | Synthetic — the repository's MIT license applies |
| Size (committed) | `warehouse_scene.obj` ~68 KiB, `materials.csv` ~0.8 KiB, `sensor_config.csv` ~0.5 KiB, `sensor_poses.csv` ~0.3 KiB — ~70 KiB total |
| Checksums (SHA-256) | `warehouse_scene.obj` = `5b250eadb28fae9ded130e64693a90e286fc4e9f7c0ea072249b420dba3958d5`<br>`materials.csv` = `ead8d50c16c71779807b549afa613df0525f3264316e2bb3d24fc109c944533d`<br>`sensor_config.csv` = `bdaae15c1e1db5721f45191d4e2e2a23035e90cc6ff5207784722f5f0ba2196d`<br>`sensor_poses.csv` = `7506b42fb1a426f74c3c0e070c3b7d1feaff00a3f8d12abec6a905ab052efd16` |

No public dataset applies here (see `../scripts/download_data.ps1`/`.sh`, both honest no-ops): no public
corpus ships a synthetic-LiDAR scene *with* the exact analytic ground truth (a flat, exactly-flat floor;
known material albedos) this project's verification gates need — see `../src/main.cu`'s ANALYTIC GATES
stage and [`../THEORY.md`](../THEORY.md) "How we verify correctness".

### Fields / format

**`warehouse_scene.obj`** — a hand-rolled-loader-friendly Wavefront OBJ: only `v x y z` (vertex, meters,
world frame, right-handed, z-up) and `f a b c` (triangle face, bare 1-indexed integers — no texture/
normal indices) lines, plus `#`-prefixed comments. **2,264 triangles, 1,233 vertices.** Face emission
order is fixed and load-bearing: ground grid (32x32 = 2,048 triangles), then 4 perimeter walls (48
triangles), then 6 shelving racks (72 triangles), then 8 crates (96 triangles) — `materials.csv`'s
`RANGE` rows below index into this exact order. The scene spans a 24 m x 24 m floor (`[-12, +12]` m in
x and y), 3 m walls, and keeps the `+x` corridor near the origin (`y ~ 0`) clear of obstacles out to the
east wall — [`../src/main.cu`](../src/main.cu)'s analytic verification gates fire single test beams
down that corridor and need a guaranteed line of sight to the floor.

**`materials.csv`** — per-triangle-index-range material assignment (not a full OBJ+MTL parser —
CLAUDE.md §5's "hand-roll rather than link a library" stance, and the scene generator controls face
order exactly, so a small range table is simpler and equally correct):

| Row type | Fields | Meaning |
|---|---|---|
| `MATERIAL,id,albedo,name` | `id` (0-indexed, in order), `albedo` (dimensionless Lambertian reflectance, `(0,1]`), `name` | One row per material — see [`../THEORY.md`](../THEORY.md) "The problem" for what albedo means physically |
| `RANGE,tri_start,tri_end,material_id` | inclusive, 0-indexed triangle range (face order in the OBJ above) | Every triangle is covered by exactly one `RANGE` row; the loader in `../src/main.cu` aborts if any triangle is uncovered or double-covered |

Committed materials: `0` concrete floor (albedo 0.30), `1` painted steel wall (0.55), `2` steel shelving
(0.65), `3` cardboard crate (0.80) — illustrative Lambertian reflectance values in the real-world range
for these surface types (THEORY.md cites orders of magnitude, not a specific measured target).

**`sensor_config.csv`** — `LABEL,value` rows (angles in degrees/milliradians as labeled; the loader
converts to radians once). Fields map directly onto `SensorConfig` in
[`../src/kernels.cuh`](../src/kernels.cuh) — read that struct's comments for the physical meaning of
every field (scan pattern: `CHANNELS`, `AZIMUTH_STEPS`, `ELEVATION_MIN/MAX_DEG`, `AZIMUTH_START_DEG`,
`RANGE_MIN/MAX_M`; beam divergence: `DIVERGENCE_HALF_ANGLE_MRAD`, `SUBRAY_COUNT`; intensity radiometry:
`INTENSITY_GAIN`; dropout + range noise: `RANGE_NOISE_BASE/PER_M`, `DROPOUT_BASE`,
`DROPOUT_RANGE_COEFF`, `DROPOUT_INCIDENCE_COEFF`, `SEED`). Committed values: 32 channels x 1,024
azimuth steps = 32,768 beams/frame (exactly the catalog bullet's own "32 x 1024 = 32k beams/frame"
example), elevation -15° to +15°, 1.5 mrad beam divergence with 4 jittered subrays, seed 42.

**`sensor_poses.csv`** — `POSE,frame_idx,x,y,z,qw,qx,qy,qz` rows: `T_world_sensor` (meters; unit
quaternion, **repo order `w,x,y,z`**, SYSTEM_DESIGN.md §3.4). v1 uses only `frame_idx` 0 (an AMR-style
LiDAR mounted level at 1.5 m, centered in the room, identity orientation — facing `+x`); the format
allows more frames for a future multi-frame demo (README §Exercises).

Every loader is **strict**: an unrecognized label, a short row, an uncovered triangle, or a `RANGE`
referencing an undefined material aborts the demo loudly (the repo's strict-loader convention — 08.01's
scenario loader, 02.06's cloud loader) instead of silently mis-simulating.
