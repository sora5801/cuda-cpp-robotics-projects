# Demo — 16.01 Thruster allocation for overactuated ROVs (batched QP)

One command builds (if needed), runs on the committed sample, and verifies the output:

```powershell
.\run_demo.ps1          # Windows (the required path: VS 2026 + CUDA 13.3)
```

```bash
./run_demo.sh           # Linux/macOS best-effort bonus (CMake path)
```

## What the demo demonstrates

The full pipeline described in `../README.md` and `../src/main.cu`'s file header: build the vehicle's
8-thruster allocation matrix once, load a synthetic 500-sample commanded-wrench trajectory, allocate the
whole batch on the GPU (one thread per QP) and on the CPU oracle, check the two agree (`VERIFY`), check
the result against **two independent mathematical ground truths** — the closed-form damped pseudoinverse
for unsaturated wrenches (`GATE-PSEUDOINV`) and the box-KKT fixed-point condition for saturated ones
(`GATE-KKT`) — then check the optimizer's objective is monotone non-increasing on the project's
motivating worked example (`GATE-MONOTONE`, THEORY.md walks the exact numbers). Finally it writes two
CSV artifacts and prints a nine-configuration thruster-failure sweep.

**Artifacts** (written to `out/`, git-ignored — regenerated by every run):

- `out/allocation.csv` — one row per commanded wrench in the batch: commanded wrench (6 columns),
  achieved wrench `B*u` (6 columns), all 8 thruster forces, and a `saturated` flag. Plot commanded vs.
  achieved `Fx`/`Mz` over `t_s` to see the two docking-correction bursts (around `t=15s` and `t=33s`)
  visibly bend the achieved wrench away from commanded once thrusters saturate.
- `out/failure_analysis.csv` — one row per configuration (`nominal`, then `thruster_0_failed` ..
  `thruster_7_failed`): mean/max relative tracking error, and — the metric that actually isolates a
  failure's effect (see the comment in `main.cu`) — mean/max **degradation relative to the nominal
  baseline**, plus a count of wrenches "significantly worse" than nominal. Compare the horizontal
  thrusters (`thruster_0..3_failed`, i.e. H1-H4) against the vertical ones (`thruster_4..7_failed`,
  V1-V4): losing a horizontal thruster degrades this batch's tracking far more, because the committed
  wrench trajectory's bursts are horizontal-group-heavy (surge+yaw corrections) — a directly readable
  consequence of which DOF the batch actually stresses.

## How to read the output

| Line prefix | Meaning | Checked against `expected_output.txt`? |
|-------------|---------|----------------------------------------|
| `[demo]`    | Which project/demo this is. | Yes — stable. |
| `[info]`    | GPU name, file paths, and per-stage diagnostic numbers (worst deviations, batch composition, per-configuration failure stats) — varies by machine/run. | No. |
| `PROBLEM:` / `SCENARIO:` | The problem instance (thruster/DOF counts, QP hyperparameters) and the loaded wrench batch's size. | Yes — stable. |
| `[time]`    | CPU vs. GPU kernel timing for the whole 500-QP batch — a **teaching artifact, never a benchmark claim** (single-shot, kernel-only). | No. |
| `VERIFY:`   | GPU-vs-CPU agreement verdict (the §5 gate). | Yes — stable. |
| `GATE-PSEUDOINV:` / `GATE-KKT:` / `GATE-MONOTONE:` | The three optimality verdicts (THEORY.md "How we verify correctness"). | Yes — stable. |
| `ARTIFACT:` | Confirms each CSV was written (or reports a write failure). | Yes — stable. |
| `RESULT:`   | Overall `PASS`/`FAIL`. The program exits nonzero on `FAIL`. | Yes — stable. |

The runner scripts do a **subset diff**: every non-comment line of
[`expected_output.txt`](expected_output.txt) must appear verbatim in the output; extra lines
(timings, device info, per-stage `[info]` diagnostics) are allowed. `#`-prefixed lines in that file are
comments.

## If it fails

- **Build fails:** see [`../../../docs/BUILD_GUIDE.md`](../../../docs/BUILD_GUIDE.md) (toolchain
  install, CUDA/VS integration, GPU architecture list).
- **`VERIFY: FAIL`:** the GPU result disagreed with the CPU oracle — a real bug. Start in
  `../src/kernels.cu` and compare against `../src/reference_cpu.cpp`'s `thruster_allocate_cpu`.
- **`GATE-PSEUDOINV: FAIL` or `GATE-KKT: FAIL`:** the optimizer converged to the wrong answer even
  though GPU and CPU agree with each other — check `kPgdIters`/`step` in `kernels.cuh` (THEORY.md "the
  math" derives the Lipschitz-safe step size) or the QP formulation itself in `build_qp_matrices`.
- **`GATE-MONOTONE: FAIL`:** the projected-gradient step size exceeds `1/L` somewhere (a genuine descent
  violation, not FP32 noise) — see THEORY.md "numerical considerations" for the documented FP32 slack
  and how it was measured.
- **Expected-line mismatch only:** the program passed its own checks but printed different stable
  lines — someone changed the output without updating `expected_output.txt` (or vice versa). The two
  are a contract; fix them together.
