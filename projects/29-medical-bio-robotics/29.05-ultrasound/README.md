## Overview

This project builds a complete, GPU-accelerated **B-mode ultrasound beamformer**: it simulates
the raw echo data a 64-element linear array would record from a synthetic phantom, then
reconstructs a 2-D image from that data using the same three-stage pipeline real "software
beamformers" use — **delay-and-sum (DAS)**, **quadrature-demodulation envelope detection**, and
**log compression**. The catalog bullet bundles three milestones (CLAUDE.md §2); this project
implements the first **completely**:

- **Implemented — Milestone 1, beamforming:** the full pixel-parallel DAS pipeline below, verified
  against a CPU oracle at every stage and against four independent, physics-derived ground-truth
  checks (wire-target localization, measured-vs-derived resolution, inclusion contrast, and a
  closed-form delay sanity check).
- **Documented only — Milestone 2, elastography:** estimating tissue *stiffness* from small
  induced displacements between successive beamformed frames (strain imaging / shear-wave
  elastography). [THEORY.md §Where this sits in the real world](THEORY.md#where-this-sits-in-the-real-world)
  describes the method and what this project's B-mode pipeline would feed it.
- **Documented only — Milestone 3, image-based servoing:** closing a robot-arm control loop on
  live B-mode frames (steering a probe to keep an anatomical target centered, or scanning a
  path). [README §System context](#system-context--where-this-sits-in-a-robot) and THEORY.md
  describe the loop this project's frame rate would need to close.

A learner who studies this project understands: why ultrasound speckle is *interference*, not
noise; how a modern "ultrafast" scanner reconstructs a whole image from a single plane-wave
transmit instead of steering a beam scanline by scanline; why every pixel of that reconstruction
is an independent, embarrassingly parallel problem (the textbook GPU-beamforming argument); and
how to turn "resolution" and "contrast" from marketing words into numbers you measure from a
simulated point-spread function and check against a derived formula.

## What this computes & why the GPU helps

Per image (206,000-ish pixels, one run — this is not a streaming demo): reconstruct every pixel
independently from 64 channel traces, then demodulate, low-pass filter, and log-compress.

- **Pattern:** pixel-parallel **map** for DAS, quadrature-demod, and log-compression (one thread
  per pixel, no interaction between pixels); a small 1-D **stencil** (17-tap FIR) along depth for
  the envelope low-pass.
- **Measured reality (this repo's reference RTX 2080 SUPER, single-shot, teaching artifact):** the
  full 4-kernel GPU pipeline runs in ~1–2 ms; the CPU oracle takes ~110–125 ms for the same
  pipeline — roughly a 60–110x speed-up on the *reconstruction*, which is exactly the DAS
  computation real ultrafast scanners must repeat every ~0.1–1 ms to hit kilohertz frame rates
  (see System context below). Channel-data *synthesis* (the phantom's photon-, sorry,
  **phonon**-level simulation) is deliberately host-only setup, not the taught algorithm — see
  [kernels.cuh](src/kernels.cuh)'s file header.
- **Why pixel-parallel, not element-parallel or scanline-parallel:** the classic beamforming GPU
  argument (THEORY.md "The GPU mapping" derives it in full) — every output pixel needs its own
  small, independent per-element loop, so the natural mapping puts one GPU thread on the biggest
  axis (pixels, ~206k), not the smallest (elements, 64).

## System context — where this sits in a robot

Where this project lives in the canonical autonomy stack (see
[`../../../docs/SYSTEM_DESIGN.md`](../../../docs/SYSTEM_DESIGN.md)) and in the physical/commercial whole
(see [`PRACTICE.md`](PRACTICE.md)).

- **Stack position:** the **perception front end** for an image-guided medical/robotic system —
  it sits between the raw sensor (the transducer array) and everything downstream that consumes
  an *image* rather than raw RF. In SYSTEM_DESIGN.md §1's stack terms it is SENSORS → this
  project's PERCEPTION box; elastography (milestone 2) is the next perception stage, and
  image-based servoing (milestone 3) is the CONTROL box that closes the loop back onto the
  probe-holding arm.
- **Upstream inputs:** the transducer array's raw per-element RF traces (this project's
  `channel_data[kNumElements][kNumSamples]` — the message-shaped equivalent of a raw ADC capture,
  analogous to a `sensor_msgs/Image`'s raw Bayer buffer before debayering) — and, on a real
  system, the **robot arm holding the probe** (image-guided robotics: the arm's pose tells the
  beamformer where its image sits in the robot's or patient's frame, `T_base_probe` in this
  repo's transform convention).
- **Downstream consumers:** the two documented-only milestones, **by name**: **elastography**
  (milestone 2 — consumes a *sequence* of this project's B-mode frames, tracks sub-resolution
  displacement between them, and outputs a stiffness map) and **image-based servoing**
  (milestone 3 — consumes a *live stream* of this project's B-mode frames as the feedback signal
  for a visual servo loop that moves the probe-holding arm, the same `image → feature error →
  joint velocity` shape as 21.x's visual servoing projects, just with a B-mode frame instead of a
  camera frame as the sensor).
- **Rate / latency budget:** this is the project's sharpest robotics tie. **Plane-wave
  "ultrafast" imaging — the transmit scheme this project uses — exists specifically because
  reconstructing a whole frame from ONE transmit (instead of ~64–128 focused scanline transmits)
  unlocks kilohertz raw frame rates**, where conventional focused-beam scanning tops out around
  30–60 Hz (the same 30–60 Hz "camera → perception" budget in SYSTEM_DESIGN.md §1.1's table).
  Image-based servoing needs that headroom: a visual servo loop wants to run near the "local
  planner / trajectory replan" band (10–50 Hz, SYSTEM_DESIGN.md §1.1) at minimum, and a
  tele-operated or contact-sensitive probe-steering loop benefits from the higher end of that —
  which only plane-wave-class frame rates can deliver with headroom to spare. This project's own
  GPU reconstruction (~1–2 ms measured) is not the bottleneck; the physics of acquiring a new
  plane-wave frame (a few pulse round-trips deep into tissue, tens of microseconds to a few
  milliseconds depending on depth) is.
- **Reference robot(s):** none of SYSTEM_DESIGN.md §2's five reference robots directly (they are
  warehouse AMR, 6-DoF manipulator cell, quadruped, quadrotor, and AV stack — all non-medical),
  the same honest gap 17.01 (space) names for its domain. This project's "robot" is its own
  archetype: a **robot-held ultrasound probe / image-guided intervention system** — closest in
  *shape* to the manipulator work cell (SYSTEM_DESIGN.md §2.2: a precise arm doing sensor-guided
  motion in a constrained cell) but its own domain, with its own PRACTICE.md-grounded hardware
  and regulatory reality.
- **In production:** research/clinical platforms expose exactly this pipeline as a programmable
  API (Verasonics Vantage — see "Prior art"); k-Wave and Field II simulate the acoustics this
  project's channel-data synthesizer approximates; the PICMUS challenge is the public benchmark
  this project's plane-wave/DAS design choices are modeled on.
- **Owning team:** medical imaging / robotics R&D (SYSTEM_DESIGN.md item 5's "controls & autonomy"
  and "ML/data" teams, specialized to a medical-device product) — with regulatory/compliance
  holding veto power the moment any of this touches a real patient (PRACTICE.md §4).

## The algorithm in brief

- **Plane-wave transmit, pixel-based DAS receive** — one unsteered plane wave insonifies the
  whole field of view; every pixel is reconstructed independently by summing every element's
  channel sample at that pixel's own two-way delay. → [THEORY.md §The math](THEORY.md#the-math),
  [THEORY.md §The algorithm](THEORY.md#the-algorithm).
- **F-number-limited Hann apodization** — one continuous formula both grows the active receive
  aperture with depth and tapers it smoothly, trading resolution against sidelobes with a single
  knob. → THEORY.md §The algorithm.
- **Linear delay interpolation** — the fractional-sample two-way delay is interpolated, not
  rounded, or the reconstructed image rings. → THEORY.md §Numerical considerations.
- **Hilbert-free quadrature-demodulation envelope detection** — mix with the carrier, then a
  small Hann-windowed-sinc FIR low-pass, then magnitude — the narrowband-signal substitute for a
  full Hilbert transform. → THEORY.md §The algorithm.
- **Log compression** — 20·log10(envelope/max), clamped to a 50 dB display range — the standard
  reason ultrasound images are always shown in decibels. → THEORY.md §The problem.

## Build

Requires Visual Studio 2026 (v145 toolset) + CUDA Toolkit 13.3 — full install and troubleshooting steps
live in [`../../../docs/BUILD_GUIDE.md`](../../../docs/BUILD_GUIDE.md).

1. Open [`build/ultrasound.sln`](build/ultrasound.sln) in Visual Studio 2026.
2. Select the `Release|x64` configuration.
3. Build (Ctrl+Shift+B). The executable lands at `build/x64/Release/ultrasound.exe`.

Optional cross-platform path: `CMakeLists.txt` at the project root (a bonus for Linux learners; the VS
solution is the required deliverable, CLAUDE.md §5).

Optional dependencies and their fallbacks: **none** — CUDA runtime + C++17 standard library only.
No cuFFT/cuBLAS/Thrust: DAS is a geometric sum with interpolation (not a transform), and the
envelope's low-pass is a small enough FIR that a hand-rolled stencil kernel teaches more than a
library call would (CLAUDE.md §5).

## Expected output

Ten stable lines — banner, `PROBLEM:`, `SCENARIO:`, `VERIFY:`, `LOCALIZATION:`, `RESOLUTION:`,
`CONTRAST:`, `DELAY_CHECK:`, `ARTIFACT:`, `RESULT:` — checked as a subset diff by
[`demo/expected_output.txt`](demo/expected_output.txt). **Five independent verifications**,
measured on this repo's reference machine (single-shot; magnitudes will differ modestly on other
GPUs, verdicts should not — CLAUDE.md §12):

1. **The §5 GPU-vs-CPU gate, three stages** (`VERIFY:`) — the beamformed RF image, the envelope,
   and the log-compressed dB image each compared against `src/reference_cpu.cpp`'s oracle;
   measured worst absolute deviations ~1.5e-3 (tol 0.01), ~3e-4 (tol 0.005), ~0.013 dB (tol 0.05
   dB) — see README §Limitations for why these are *absolute*, not relative, tolerances.
2. **`LOCALIZATION:`** — every one of the 9 wire targets' image peak must land within one
   resolution cell (0.462 mm) of its true position; measured worst case ~0.18 mm.
3. **`RESOLUTION:`** — the isolated center wire's measured −6 dB point-spread-function width must
   fall within a 0.3x–3.0x factor of the derived λ·F# (lateral) / cycles·λ/2 (axial) formulas;
   measured ratios ~2.5x (lateral) and ~0.8x (axial).
4. **`CONTRAST:`** — the high-scattering inclusion's mean dB must exceed background speckle's
   mean dB by at least 2 dB; measured margin ~7.6–8.7 dB.
5. **`DELAY_CHECK:`** — the DAS delay formula, evaluated for one known pixel/element pair,
   matches an independently re-derived closed-form two-way time to within 5 ns; measured
   difference ~7e-5 ns.

## Code tour

A guided reading order through `src/`:

1. [`src/kernels.cuh`](src/kernels.cuh) — the project's one-place contract: array/pulse/grid
   constants, the channel-data and image memory layouts, and the shared FIR-design function. Read
   this first — everything else restates its formulas.
2. [`src/main.cu`](src/main.cu) — orchestration: phantom loading (with the 03.01-style
   cross-check), `simulate_channel_data()` (the synthetic sensor), the GPU + CPU pipeline calls,
   the three-stage VERIFY gate, and the four ground-truth gates. The single most interesting
   thing to read here: `simulate_channel_data()`'s comment block on why coherently summing many
   equal-amplitude, randomly-positioned scatterers *is* the speckle pattern.
3. [`src/kernels.cu`](src/kernels.cu) — the four GPU kernels. Start with `das_kernel` — the
   project's star: one thread per pixel, a 64-element inner loop, and the single continuous
   apodization formula that does both f-number aperture growth and Hann tapering at once.
4. [`src/reference_cpu.cpp`](src/reference_cpu.cpp) — the line-by-line CPU twins.
5. [`src/util/`](src/util/README.md) — `CUDA_CHECK`, timers (copied, not shared — §4 rule).

## Prior art & further reading

The real tools and papers this project teaches toward — study them, do not copy them (CLAUDE.md §4.1).

- **Verasonics Vantage** — the research ultrasound platform this project's software-beamforming
  shape imitates: programmable plane-wave/DAS pipelines on real hardware, the professional version
  of exactly what `kernels.cu` does.
- **PICMUS challenge (Liu et al., IEEE TUFFC 2015)** — the public plane-wave beamforming
  benchmark this project's transmit/DAS/apodization choices are modeled on; PICMUS's simulated and
  experimental datasets are the natural next step after this project's synthetic phantom.
- **k-Wave (Treeby & Cox) / Field II (Jensen)** — full acoustic-wave and linear-systems ultrasound
  simulators; this project's Gaussian-pulse channel synthesizer is a deliberately simplified
  stand-in for what these tools compute rigorously (THEORY.md names the gap).
- **Jensen, "Linear description of ultrasound imaging systems"** — the standard reference for the
  point-spread-function/resolution formulas THEORY.md derives and this project measures.
- **GE/Philips/Siemens clinical scanners** — the commercial end state: the same three-stage
  pipeline, plus adaptive beamforming, harmonic imaging, and decades of clutter/speckle-reduction
  engineering this project's "Limitations" section names honestly as out of scope.

## Exercises

1. **Plot the artifacts:** `demo/out/bmode.pgm` (open with any image viewer that reads PGM, or
   convert with ImageMagick/PIL) and `demo/out/psf_profile.csv` (plot `db_relative` vs
   `position_mm`, filtered by `axis`) — see the −6 dB crossings the RESOLUTION gate measures.
2. **Nearest-neighbor vs. linear interpolation:** replace `das_kernel`'s linear interpolation with
   simple rounding to the nearest sample, rebuild, and look at `bmode.pgm` — THEORY.md predicts
   visible axial ringing; confirm it and explain why from the sampling rate and carrier frequency.
3. **Focused vs. plane-wave transmit:** change the transmit delay formula from `t_tx = z/c` (plane
   wave) to a focused-transmit delay converging at a chosen focal depth, and compare the resulting
   resolution/contrast at that depth vs. away from it — the classic focused-beam trade THEORY.md
   describes but this project does not implement.
4. **Shared-memory halo tiling:** `envelope_lowpass_kernel`'s FIR stencil currently re-reads
   overlapping global-memory rows across neighboring threads (kernels.cu's header comment names
   this). Add a shared-memory halo tile along depth and measure the kernel-time change.
5. **Climb toward milestone 2:** log two B-mode frames from phantoms with a small (sub-pixel)
   scatterer displacement between them, and estimate the displacement field with a simple
   block-matching cross-correlation between corresponding image patches — the first building
   block of strain elastography (THEORY.md "Where this sits in the real world").

## Limitations & honesty

- **Educational and synthetic only.** Every scatterer is numerically generated
  (`scripts/make_synthetic.py`, seed 42); nothing here is patient data, and nothing this project
  computes is a diagnostic or therapeutic claim (CLAUDE.md §1, §8). "Wire target" and "inclusion"
  are signal-processing test structures, never called a lesion or any clinical term.
- **Reduced-scope catalog bullet.** Only milestone 1 (beamforming) is implemented; elastography
  and image-based servoing are documented-only, per CLAUDE.md §2's bundled-bullet rule — see
  README §Overview and THEORY.md "Where this sits in the real world" for what each would add.
- **Absolute, not relative, GPU-vs-CPU tolerances.** An earlier version of the VERIFY gate used a
  relative-with-floor tolerance and reported a spurious 5% worst-case deviation traced to a single
  near-null pixel (the RF signal oscillates through zero and the envelope genuinely visits
  near-zero values at destructive-interference speckle nulls — dividing a tiny, real FP32
  rounding difference by an arbitrary small floor manufactures a large, meaningless ratio there).
  The gate now compares absolute deviations, which stay small and stable everywhere in the image —
  see `src/main.cu`'s VERIFY comment and THEORY.md "Numerical considerations" for the full story.
- **A simplified acoustic model.** Channel-data synthesis uses a single reflection coefficient per
  scatterer, 1/r receive-spreading loss, and no frequency-dependent tissue attenuation, no
  multiple scattering, and no aberration — real tissue is far messier (THEORY.md names what
  k-Wave/Field II model that this project does not).
- **No time-gain compensation (TGC).** DAS here normalizes by the sum of apodization weights
  rather than depth-dependent gain, a deliberate simplification documented in `kernels.cu`'s
  `das_kernel` comment (real scanners combine both).
- **No adaptive beamforming, harmonic imaging, or speckle reduction** — the commercial techniques
  named in "Prior art" that separate a research beamformer from a clinical product.
- **Timings are teaching artifacts** — single-shot, one machine (RTX 2080 SUPER), kernel-only
  where labeled (CLAUDE.md §12).
- **Sim-validated only (CLAUDE.md §1):** nothing in this project drives real hardware; if this
  pipeline ever fed a real robot-held probe or a servoing loop, the full testing ladder
  (PRACTICE.md §3) and an independent safety envelope would be required first.
