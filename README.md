# cuda-cpp-robotics-projects

**A didactic study collection: 505 self-contained CUDA C++ projects across 36 domains, spanning all of
robotics — built to teach, not to deploy.**

Every project here parallelizes a real robotics computation on the GPU — stereo depth, particle filters,
TSDF fusion, MPPI control, rigid-body dynamics, swarm simulation, battery models, soft-body FEM, and
hundreds more — and then explains itself completely: the physics, the math, the algorithm, the CUDA
mapping, the hardware it would run on, and the place it occupies inside a real robot and a real robotics
company.

The philosophy (from [CLAUDE.md](CLAUDE.md), this repo's binding contract):

- **Teaching beats cleverness.** A slower kernel a learner can follow beats a fast one they cannot —
  the fast version is explained in comments instead.
- **Every artifact must explain itself.** Code, build files, data scripts, and demos are all teaching
  surfaces. Sparse comments mean the file is unfinished, no matter how well it runs.
- **Nothing is a black box.** Every library call (cuBLAS, cuFFT, Thrust…) is annotated with what it
  computes, why it is used, and what writing it by hand would take.
- **Every project knows its place in a robot — and in the whole.** Each project states what feeds it,
  what consumes it, at what rate it must run, which reference robot uses it, and which team in a
  robotics company would own it. A kernel without system context is half-taught.
- **Reproducibility is sacred.** Clone, open the solution, build, run the demo, see the documented
  result — on a normal Windows + NVIDIA machine, with no robot hardware attached.

## Who this is for & how to study with it

This repository is for someone who knows C++ and wants to learn **CUDA and robotics together**, one
self-contained project at a time. Each project stands alone: no project depends on another at build or
run time, and every demo runs offline on a tiny committed sample.

The study loop for any project:

1. **Clone** the repo (once).
2. **Pick a project** — start from the [domain index](#domain-index) below, or from a ★-marked
   beginner entry. Open `projects/<domain>/<SS.NN>-<slug>/`.
3. **Build** — open `build/<slug>.sln` in Visual Studio 2026, select `Release|x64`, Build.
4. **Run the demo** — `demo/run_demo.ps1` (one command: runs on the sample data, prints the result,
   checks the GPU output against a CPU reference, and shows a timing line).
5. **Read, in order:**
   - the project's `README.md` — what it computes, why the GPU helps, and its **System context**
     (where it sits in a robot);
   - `THEORY.md` — physics → math → algorithm → GPU mapping → numerics → verification;
   - `PRACTICE.md` — how the physical subsystem is built, the real chips and sensors involved, how
     the code would be installed on a robot, and the business/regulatory reality around it;
   - `src/` — following the README's **Code tour** reading order (typically `main.cu` →
     `kernels.cu` → `reference_cpu.cpp`). The code is over-commented on purpose; the comments are
     the textbook.
6. **Do the exercises** at the end of the README, then pick the next project.

## Toolchain

| Component | Requirement |
|---|---|
| OS | Windows 10/11, x64 |
| IDE / compiler | **Visual Studio 2026** (Community is fine), *Desktop development with C++*, platform toolset **v145** |
| GPU toolkit | **CUDA Toolkit 13.3** with the Visual Studio MSBuild integration |
| GPU | NVIDIA, compute capability **sm_75 (Turing) or newer** — CUDA 13 dropped Maxwell/Pascal/Volta, so sm_75 is the floor |
| Optional | Python 3.12+ for the synthetic-data scripts in each project's `scripts/` |

Full installation and troubleshooting instructions — including how driver/toolkit CUDA versions relate
(minor-version compatibility within the 13.x family) and how to narrow the compiled architecture list
for faster local builds — live in [docs/BUILD_GUIDE.md](docs/BUILD_GUIDE.md).

## Anatomy of a project

Every one of the 505 projects has exactly the same shape. Each piece is a teaching surface:

| Piece | What it teaches |
|---|---|
| `README.md` | The entry point: what it computes, why the GPU helps, **where it sits in a robot**, how to build/run, a guided code tour, exercises, and honest limitations. |
| `THEORY.md` | The deep dive: physics & engineering first, then the math, the algorithm, the GPU mapping (threads/blocks/memory hierarchy), numerical pitfalls, and how correctness is verified. |
| `PRACTICE.md` | From code to machine: how the physical subsystem is constructed, the real chips/sensors/actuators involved (illustrative BOM), installation on a real robot, and business & regulatory context. |
| `src/` | Maximally-commented CUDA C++: `main.cu` (entry), `kernels.cu`/`.cuh` (the GPU kernels), `reference_cpu.cpp` (the plain-C++ oracle the GPU result is checked against), `util/` (timing, error-checking helpers). |
| `data/` | A tiny committed sample (`data/sample/`, labeled synthetic or public) so the demo runs with zero downloads, plus a `data/README.md` covering provenance and licensing. |
| `scripts/` | `make_synthetic.py` regenerates the sample with full ground truth; `download_data.*` fetches a public dataset where one genuinely teaches more. |
| `demo/` | `run_demo.ps1` — one command that builds if needed, runs on the sample, prints the labeled result plus the GPU-vs-CPU agreement check, and matches `expected_output.txt`. |
| `build/` | The required Visual Studio 2026 solution (`.sln`/`.vcxproj`, commented). An optional `CMakeLists.txt` at the project root serves Linux learners. |

## How the projects compose into robots

The projects are self-contained to build, but they are designed to **compose conceptually** into full
autonomy stacks: shared SI units and frame conventions, message-shaped structs that deliberately
resemble ROS 2 types, and a common layer diagram from sensors through perception, estimation, planning,
and control down to actuation. [docs/SYSTEM_DESIGN.md](docs/SYSTEM_DESIGN.md) is the shared architecture
reference — five reference robots (warehouse AMR, manipulator work cell, quadruped, quadrotor,
autonomous vehicle) and a **composition map** of worked chains through the repo (e.g., LiDAR sim → ICP →
TSDF fusion → distance field → STOMP planning → MPPI control) showing how study projects grow into a
robot.

## Safety & honesty

> **Educational only — not for deployment.** Everything in this repository is study material. Control
> and planning code is validated **in simulation only**. Nothing is safety-certified: no compliance
> with ISO 10218, ISO 13482, ISO/TS 15066, ISO 26262, or any other standard is claimed — even projects
> that *compute* metrics from those standards do so didactically. Running any of this on a physical
> robot is entirely the operator's decision and responsibility. Timing numbers are teaching artifacts
> from documented local runs, never benchmark claims; results are never fabricated. No project is
> framed for weaponization or for surveillance of individuals — person-perception projects are framed
> around collaborative safety.
